# ZPay 第三方聚合支付接入指南

> 记录于 2026-06-30，黄金换算助手项目实战经验。
> 个人/个体户在没有营业执照高级资质、没有腾讯云商付资质的情况下，最快可用的微信收款方案。

---

## 一、为什么选 ZPay

跑通收款这件事，原本考虑过腾讯云商付，但腾讯云商付需要营业执照 + 已备案域名，审核周期长，对项目早期验证非常不友好。

ZPay 的优势：

- **门槛低**：个体工商户即可签约，不需要等待复杂资质审核
- **支持微信扫码/H5支付**：覆盖移动端浏览器场景
- **有官方备案**：中国支付清算协会备案，相对正规
- **T+1 结算**：资金自动下发到银行卡
- **接入快**：API简单，当天就能跑通

**代价**：免不了的代价是手续费比官方渠道高，且部分支付方式（如微信H5直接唤起App）依赖渠道资质，个体户渠道默认只能用"扫码支付"，无法做到系统浏览器直接唤起微信App完成支付（除非额外购买V1接口，¥88开通费 + 更高手续费）。

---

## 二、商户后台关键信息

登录地址：`https://z-pay.cn`（会员中心 `member.z-pay.cn`）

商户信息（本项目示例，已脱敏）：
- 渠道商户号（pid）：在"我的支付渠道"页面查看
- 商户密钥（key）：在渠道详情页查看，**仅显示一次/需妥善保管**
- 接口文档：`https://z-pay.cn/doc.html`

**资料准备（开户用）**：
1. 身份证正反面照片
2. **不需要营业执照**，个人站长也可申请
3. 每个渠道开户费88元起，开户失败全额退款

## 三、核心接口

### 1. 创建订单（mapi.php）

```
POST https://zpayz.cn/mapi.php
```

参数（MD5签名）：
```js
{
  pid,              // 商户ID
  type,             // 'wxpay' 微信 / 'alipay' 支付宝
  out_trade_no,     // 商户订单号，自己生成唯一值
  notify_url,       // 异步回调地址
  return_url,       // 同步跳转地址（支付完成后用户停留页）
  name,             // 商品名称，必须真实具体，否则容易被封
  money,            // 金额，最多两位小数
  clientip,         // 用户IP
  device: 'h5',     // 'h5' 用于网页端跳转支付（注意不是 'mobile'）
  param,            // 自定义透传参数，回调时原样返回
  sign_type: 'MD5',
  sign,             // 签名
}
```

返回结果里：
- `payurl`：支付宝支付跳转链接
- `payurl2`：微信H5支付跳转链接（**注意优先用这个，不是 payurl**）

### 2. 异步回调（notify_url）

ZPay 用 **GET** 请求回调，关键字段：
```
trade_status   // 'TRADE_SUCCESS' 表示支付成功
out_trade_no   // 对应订单号
money          // 实际支付金额
param          // 创建订单时传的透传参数，原样返回
sign           // 签名，需要验证
```

**收到回调后必须返回字符串 `"success"`，否则ZPay会重复推送回调。**

## 四、踩过的坑

### 坑1：device 参数选错，导致微信内置浏览器才能打开支付页

最初用 `device: 'mobile'`，结果用户在系统浏览器（Safari/Chrome）里点击支付，会跳到一个提示"请在微信客户端打开链接"的页面，根本付不了钱。

**解决**：`device` 改为 `'h5'`，并优先取返回结果的 `payurl2` 字段（而不是 `payurl`），这才是真正的微信H5支付链接，可以在系统浏览器里唤起微信App完成支付。

**但这一步之所以最终能跑通，前提是开通了正确的接口套餐——这是本项目耗时最长的一个坑，单独在坑2里详细记录。**

### 坑2：微信支付三档接口选错，导致H5支付始终无法实现（耗时最长的坑）

ZPay 后台「支付渠道」里，微信支付实际分**三档接口**，价格和能力差异很大：

| 套餐 | 开户费 | 是否需要营业执照 | 支持的支付方式 |
|------|--------|------------------|----------------|
| **微信支付V1接口** | ¥88 | 需要 | 仅 扫码 + JSAPI（**不支持H5**）|
| **微信支付V2接口** | ¥288 | 有无营业执照均可 | 扫码 + JSAPI + **H5支付** |
| 支付宝V1接口 | ¥88 | 有无营业执照均可 | 电脑端 + 手机端支付（客服称支持H5，本项目未实测）|

**最初的误区**：以为只要传对 `device: 'h5'` 参数、取对 `payurl2` 字段，代码层面就能实现H5跳转支付。实际上无论代码怎么调，**渠道本身不支持H5的话，参数传得再对也没用**——V1接口（¥88）从产品形态上就只给"扫码+JSAPI"，不会返回真正可用的H5支付链接。

**这个坑前后测试排查了很长时间，反复怀疑是代码、签名、参数的问题，最后才确认是渠道套餐本身的限制。**

**结论**：如果业务场景需要"用户在系统浏览器里点击支付按钮、自动唤起微信App完成支付"（即H5支付），**必须开通V2接口（¥288）**，V1接口（¥88）无论如何都做不到，这是产品形态限制，不是代码问题。

本项目最终开通的是V2接口，才真正实现了系统浏览器内点击支付直接唤起微信App的体验。

### 坑3：Cloudflare Pages Functions 不支持 node:crypto

最初用 `import { createHash } from 'node:crypto'` 做MD5签名，本地测试没问题，部署到 Cloudflare Pages Functions 后报错——**Cloudflare Workers运行时不支持 Node.js 内置模块**。

**解决**：自己写纯JS实现的MD5算法（无第三方依赖），文件见本项目 `functions/api/_md5.js`，已验证可用，可直接复用到任何 Cloudflare Workers/Pages项目。

### 坑4：用户支付完成后无法自动跳回浏览器

微信支付完成后，无论 `return_url` 怎么设置，都无法做到从微信自动跳回 Safari/系统浏览器——这是 **iOS 系统级限制**，网页没有权限控制App切换，原生App才能用 Universal Link 做到。

**实际验证过的结论**：
- ZPay 的"返回商家"按钮点击后，会跳到 **ZPay自己的支付成功提示页**（不是商户设置的 `return_url`），提示"请关闭页面返回浏览器"
- 用户点击"关闭"后，回到的是**调起支付的来源**（微信聊天/文件传输助手），不是直接回到Safari
- 用户必须**手动切换App**回到Safari，这一步无法绕过

**应对方案**：前端监听 `visibilitychange` 事件，用户从微信切回浏览器时，页面从后台恢复会自动触发，借此机会主动查询支付结果，不需要用户手动刷新或点击任何按钮。

```js
document.addEventListener('visibilitychange', function() {
  if (document.visibilityState === 'visible' && localStorage.getItem('pending_order')) {
    checkPaymentResult();
  }
});
```

### 坑5（已解决）：签名验证一直失败，排查了两轮才找到真正原因

最初代码里实现了完整的签名验证逻辑，但回调时计算出的签名与ZPay发来的 `sign` 字段始终不一致，一度怀疑是参数过滤规则或字段顺序问题，生产环境曾**临时跳过签名验证**（直接信任 `trade_status === 'TRADE_SUCCESS'`），属于已知技术债。

后续排查分两轮才彻底解决：

**第一轮：`_md5.js` 自实现算法本身有bug。**
构建消息字数组时用了稀疏数组（`const x = []`），未赋值的下标是 `undefined` 而非 `0`。核心轮函数里 `undefined + 轮常量t` 会产生 `NaN`，经 `&0xFFFFFFFF` 处理后被强制归零——不仅把消息字当成了0（这是对的），**连轮常量本身也被一并吞掉**。这个bug从最简单输入（如 `md5("a")`）开始就会得到完全错误的结果，用Node内置 `crypto` 或 `spark-md5` 交叉验证几个测试字符串就能立刻发现。

修复方法：构建数组前按消息总分块数预先用 `0` 填满整个数组（`new Array(totalWords).fill(0)`），不要留空洞。

**第二轮：MD5修好后，签名依然偶发失败，最终定位是商户密钥不一致。**
`_md5.js` 修好后本以为万事大吉，结果测试时ZPay后台仍然显示"未通知成功"。KV调试日志里查不到任何 `bad_sign`/`money_mismatch` 记录（说明是运行时异常崩溃，Cloudflare返回了通用错误页而非 `success` 字符串），补上了全局 `try-catch` 后异常也没有再出现，但签名还是偶发对不上。

由于ZPay后台的商户密钥是**加密存储、无法查看明文**，Cloudflare环境变量里的密钥也是**加密的Secret类型、无法核对**，两边完全没法直接比对是否一致（比如复制粘贴时是否带入了不可见的空格或换行符）。最终采用最干净的做法：**直接在ZPay后台重置商户密钥**，拿到新密钥明文后，同步更新Cloudflare Pages的 `ZPAY_KEY` 环境变量，两边保证100%一致，问题彻底解决。

**经验总结**：
- 如果自己实现加密/哈希算法（尤其是无第三方依赖的场景），务必用官方库或语言内置实现做**多组测试用例**的交叉验证，不要只测试一次就假设正确——短字符串、含中文字符串、空字符串都容易暴露边界bug
- 涉及密钥比对类问题，如果两端都是加密存储无法互相核对明文，**重置密钥同步双端**往往比反复猜测拼接细节更快更彻底
- Cloudflare Pages Functions 的 Observability/Workers Logs 功能默认未启用，需要在 wrangler 配置里显式设置 `observability.enabled = true` 才能看到运行时日志，排查前最好先确认这个功能是否真的在收集数据（可以用一个必然会触发的请求测试），避免"看不到日志"被误判为"没有收到请求"

## 五、本项目最终业务设计

```
用户选套餐 → 前端POST /api/pay（携带设备指纹）
→ 后端用MD5签名调用ZPay mapi.php，订单透传指纹
→ 返回 payurl2，前端跳转
→ 用户在微信完成支付
→ ZPay GET请求回调 /api/notify
→ 后端解析param拿到指纹，写入KV会员记录
→ 用户手动切回浏览器，visibilitychange触发查询
→ 前端轮询 /api/result，拿到结果后强制要求手机号登录绑定
```

后续手机号登录方案上线后，订单的 `param` 字段同时透传指纹，作为登录前的过渡兜底。
