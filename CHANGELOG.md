## V 1.8.22 · 20260701

**优化：微信内强制跳转引导（`wechat-mask`）**

这是早期就存在的一个独立设计——检测到在微信内打开时，全屏强制拦截，引导用户跳出微信，因为1）支付必须在微信生态外的H5才能完成，2）产品定位是加到桌面常用，不是微信内一次性访问。这次不改设计初衷，只优化三处执行细节：

1. **文字与图标靠拢**：原先用`justify-content:space-between`+文字`flex:1`把文字和图标强行拉开到容器两端，文字内容短的时候中间空出一大段距离。改成`flex-end`+去掉`flex:1`，两者紧挨在一起
2. **文案更精确**：
   - 第二步"点击浏览器打开"改为"点击「在浏览器打开」"，直接对应微信弹出菜单里的真实按钮文字，不需要用户自己脑内翻译一遍
   - 底部"在手机浏览器中打开，体验完整功能"改为"需要在手机浏览器中打开才能使用"——原文案语气偏"可选升级"，但实际是无法关闭的强制跳转，容易让用户误以为不跳出去也能凑合看，新文案更笃定地说明这是必经步骤
3. **箭头改为更容错的方向+动态效果**："···"是微信自己原生工具栏上的元素，不在页面DOM里，拿不到真实像素坐标，只能凭经验估算，且会因设备/微信版本/状态栏高度/系统翻译工具栏等因素叠加产生偏差（实测中就遇到过因为系统翻译栏顶开导致箭头指向偏移的情况）。箭头改得更长、弧度更缓，传达"往这个方向看"的大致指向而非精确戳向一点，减少因位置估算偏差导致"指不准"的观感；同时复用`.pwa-bounce`动画让箭头动起来——此前调研`philfung/add-to-homescreen`开源库时，作者原话强调"动态箭头会显著影响转化率"。

## V 1.8.20 · 20260701

**机制修复：页脚版本号从"手写死"改为运行时自动读取CHANGELOG.md，彻底解决反复遗漏同步的问题**

V1.8.9到V1.8.18连续9个版本，页脚显示的版本号字符串一直忘了同步更新，被误判成缓存/部署问题排查了很久（V1.8.19才修正）。讨论"怎么从机制上根治，而不是继续指望自己更细心"之后，确定方向：**让页脚不再需要任何人工同步，直接在页面加载时读取`CHANGELOG.md`，自动解析出最新版本号显示出来**。

选择这个方案而不是"用Cloudflare Pages Functions动态注入Git提交号"（更彻底但要把纯静态架构改成每次请求都过一次函数，工程量更大），是因为`CHANGELOG.md`本身从头到尾从未漏过更新，是这个项目里唯一真正靠谱的"版本真相来源"——与其继续维护一个容易被遗忘的独立字符串，不如直接把页脚绑定到这个已经证明可靠的地方。

**实现**：
- `index.html`新增一段脚本，页面加载时`fetch('/CHANGELOG.md')`，用正则解析开头第一条`## V X.X.X · YYYYMMDD`格式的条目，提取版本号和日期填入页脚。已用真实CHANGELOG内容验证解析结果准确（`V 1.8.19 · 0701`）。取不到或解析失败时静默保留HTML里的兜底文字，不影响页面其他功能
- `_headers`新增`/CHANGELOG.md`的`Cache-Control: no-cache`规则——这一步是必须的，否则"自动读取"这套机制本身可能被缓存卡住，重蹈之前版本号显示落后的覆辙

以后所有版本发布，只要按现有习惯正常更新`CHANGELOG.md`顶部条目（这一步从未漏过），页脚就会自动跟着对，不再需要额外记住"同步页脚"这个动作。`sw.js`的`CACHE`常量仍需人工递增（Service Worker机制本身要求文件字节必须真实变化才能触发更新检测，无法自动化），这是目前唯一还需要人工维护的版本标记。

### 本次排查的经验总结（写入仓库存档，供以后参考）

**1. 每次改动`index.html`/`sw.js`，必须同步更新三处版本标记：`sw.js`的`CACHE`常量、`index.html`页脚显示字符串、`CHANGELOG.md`。**

这次事故的直接原因，就是每次发版时只记得递增`sw.js`的缓存版本号（这个环节因为V1.8.9修过一次bug、印象深刻，所以一直没漏），却持续漏掉了页脚显示这个独立的字符串——两处版本号写在代码的不同位置，互相之间没有任何自动同步机制，光靠"记得改"是不可靠的。

**2. 遇到"用户端显示的版本落后于预期"这类问题，第一步永远是先去源码仓库直接核对实际内容，而不是先假设是缓存或部署问题。**

本次排查绕了一大圈——怀疑浏览器缓存、换无痕模式、换网络环境、怀疑CDN边缘节点缓存、研究`_headers`配置、怀疑Cloudflare部署静默失败、逐条核对Deployments记录——这些猜测本身的排查方法和推理过程都是合理的，但排查顺序不对：应该最先做的，是最简单、成本最低的一步——直接去GitHub核对源码里写的到底是什么，而不是从最复杂的可能性开始排查。这次因为顺序颠倒，让原本几分钟能定位的问题变成了一场跨越多轮对话的排查，也让用户配合做了不少其实不必要的操作（登录Cloudflare、清缓存、查部署记录等）。

**3. CSS防溢出类bug的根因，往往是"容器宽度限制按错误的字体假设估算"，而非"要不要用clamp()流式字体"。**

本轮修复的标题截断（`max-width:50vw`按拉丁窄体字Bebas Neue估算，未考虑中文会用更宽的后备字体渲染）、滑块把手溢出（8px留白按视觉经验设置，未按把手实际半径18-22px反推）两个真实bug，共同的根源都是这一类问题。排查同类问题时，应优先核实"限制数值是否按真实渲染尺寸计算得出"，而不是纠结"是否该用流式字体"。

**4. 该项目使用纯`.pages.dev`子域名（无自定义域名/zone），Cloudflare Dashboard的"Caching → Purge Everything"功能不适用（该功能挂在zone层级下，纯pages.dev部署没有对应的zone）。**

排查缓存问题时曾尝试走这条路径，Domains列表显示为空，才确认这条路径本身对这个项目的部署方式不适用，不是操作有误。

**5. Claude沙盒环境里的`bash_tool`/`web_fetch`工具的网络访问范围，不受claude.ai设置里"Capabilities → Code execution and file creation → Domain allowlist"这个开关影响，两者是完全独立的系统。**

调整该白名单设置对这两个工具的访问权限没有任何帮助，排查网络访问问题时不要混淆这两套机制。

**修复：CDN边缘节点缓存导致部署后用户长时间看不到最新版本（结构性问题）**

实测发现，用户设备上显示的版本号一直停留在V1.8.9，而当时已经推送到V1.8.17——即使`sw.js`自身的版本号递增逻辑（V1.8.9引入）完全正确，用户依然可能长时间看不到更新。

**需要人工操作**：这次改动只能防止未来再发生同样的问题，无法追溯清除已经存在于各边缘节点的旧版本缓存——需要手动登录Cloudflare Dashboard，进入对应zone的 Caching → Configuration → Purge Everything，一次性清空当前已经存在的所有旧缓存。

## V 1.8.8 · 20260701

**服务端会话Cookie架构（会员留存与身份持久化技术方案 · 第一阶段）**

排查"用户丢失本地状态被误判为新用户"这类问题时发现根源：现有架构完全依赖 `localStorage`（会员状态、设备指纹）判断身份，但localStorage会被Safari的ITP机制在网站7天无访问后自动清空（用户完全无感知，不需要任何操作触发），而设备指纹本身也不稳定（部分浏览器出于隐私保护给Canvas渲染加噪声）。查证确认：Safari的7天清理规则**只针对localStorage和"用前端JS写入的Cookie"**，服务端通过HTTP响应头 `Set-Cookie` 直接下发的Cookie不受此限制，可设置长达400天有效期，这是WebKit工程师官方澄清过的规则。本项目的Cloudflare Pages Functions与前端页面完全同源，不存在导致该规则失效的例外情况（CNAME指向第三方/IP不匹配）。

**新增**
- `functions/api/_session.js`：新增会话Cookie辅助模块（生成加密安全的随机会话ID、解析/构建Cookie、创建手机号对应的会话记录）。会话Cookie本身只存随机ID，不直接携带手机号/会员信息（Cookie设为HttpOnly，前端JS也读不到），`session:{sessionId}` 只是指向 `phone:{phone}` 的轻量指针，会员到期时间等真实数据始终只有 `phone:{phone}` 这一份来源，避免出现两份数据不一致

**修改**
- `member.js`：查询会员状态时优先读取请求携带的会话Cookie（`session:{sessionId}` → `phone:{phone}`），只有没有Cookie或Cookie对应记录查无结果/已失效时，才回退到原有的设备指纹查询逻辑（覆盖用户刚支付成功、尚未完成手机号登录绑定这一步的过渡场景）。会话有效时会在响应里续期Cookie（滑动过期）
- `sms-bind.js`：手机号绑定成功后（含首次绑定、以及订单已绑定过的幂等返回分支），下发长效会话Cookie
- `sms-verify.js`：普通登录成功、以及"订单已绑定过"直接返回真实状态这两个分支，同样下发会话Cookie

会员到期时间等业务逻辑（V1.8.7的叠加计算）完全不受影响，本次改动只影响"如何判断当前设备对应哪个会员"这一层身份识别逻辑，是在原有设备指纹机制之上新增一条更可靠的判断路径，而非替换。

## V 1.8.7 · 20260701

**会员到期时间叠加逻辑（防止二次购买覆盖缩短会员权益 + 修复续费用户被误拦截）**

排查起因：讨论"用户重复付费会有什么问题"时发现，`sms-bind.js`原先是无条件用新订单的到期时间**覆盖**手机号原有记录——如果用户已经是会员（比如还剩700天），又不小心/有意重新购买了一个较短套餐（比如30天月卡），会员到期时间会被直接覆盖缩短，而不只是"多花冤枉钱"。

进一步排查发现一个更直接会挡住转化的bug：`sms-verify.js`原本的判断顺序是**先查手机号是否已有记录**，只要有记录（不管还在有效期内、还是已经过期）就直接返回，完全不检查这次请求带的`orderNo`（新订单号）。也就是说，一个老会员到期后重新付费续费，走"手机号登录"这一步时，会被直接拦截返回"expired"，新买的这笔订单根本没机会被核验和激活。

**修复内容：**
- `sms-verify.js`：调整判断顺序，只要请求带了`orderNo`就优先核验这笔新订单、签发绑定token，不再看手机号原来的状态；只有"普通登录"（不带订单号）才走查手机号记录的老逻辑。同时增加订单幂等检查——如果这笔订单已经被标记为`bound`（绑定过），不再重复签发token，直接返回手机号当前真实状态
- `sms-bind.js`：真正落地叠加计算：`新到期时间 = max(现有到期时间, 现在时间) + 本次购买天数`。无论是"过期后重新买"（现有到期时间已过去，max取"现在"，等价于重新计算，行为不变）、"有效期内提前续费"（在原到期时间基础上完整叠加，不会因为还没到期而吃亏）还是"第一次购买"（无现有记录，同样等价于从现在开始算），这一条公式统一处理，不需要额外分支。绑定成功后把订单标记为`bound:true`；同时增加了针对"用户网络卡顿手滑连点两次登录导致同一笔订单签发多个token、被并发的两次绑定请求重复叠加"这种竞态场景的二次防护——真正写入前会再检查一次订单是否已绑定，避免同一笔购买被叠加计算两次
- `notify.js`：写入设备指纹兜底记录（`member:{fp}`）时同步应用同样的叠加公式，保持"手机号绑定"和"指纹自动识别"两条路径行为一致。`order:{out_trade_no}`订单记录本身的`expiry`字段保留为这笔购买单独的面值到期时间（用于审计/历史展示），新增`bound`字段跟踪绑定状态；支付成功后前端轮询看到的`result`记录改为展示叠加后的真实到期时间，而不是单笔购买的面值，避免用户以为自己"只买到了这么多天"

## V 1.8.4 · 20260630

**排查修复：ZPay回调"未通知成功"问题**
- 现象：ZPay商户后台多次显示"未通知成功，请检查您的程序代码（需返回字符串success）"，KV中却查不到 `debug:bad_sign`、`debug:money_mismatch`、`order:` 等任何相关记录，说明请求执行到了某个中间步骤后异常崩溃，Cloudflare返回了通用错误页而非预期的字符串响应
- 根因定位：`notify.js` 中签名验证和金额校验两步有 try-catch 保护，但"生成激活码 → 写入会员记录 → 写入订单记录 → 写入轮询结果"这几步KV写入操作此前完全没有异常保护，一旦其中任意一步抛出未预料异常，函数会直接崩溃退出，Cloudflare 返回通用错误页，不是约定的 `success` 字符串，导致 ZPay 判定通知失败并按其重试策略（0/15/15/30/180/1800...秒）反复重试
- 修复：给整个 `onRequestGet` 处理逻辑包裹全局 try-catch，任何未预料异常都会被捕获并写入 `debug:crash:{timestamp}`（包含完整错误信息与堆栈），可直接在KV调试面板查看，不必依赖 Cloudflare Observability（该功能需要在 wrangler 配置中显式开启，本项目当前未启用，暂不作为排查手段）
- 排查过程中确认：Cloudflare Pages Functions 的 Observability/Workers Logs 功能默认未开启，即便有正常流量也不会产生任何日志事件，不能用来判断请求是否到达

## V 1.8.3 · 20260630

**安全修复（严重，根因修复）**
- 修复 `_md5.js` 自实现MD5算法中的根本性bug：构建消息字数组时使用稀疏数组（`const x = []`），未被显式赋值的下标为 `undefined` 而非 `0`。在核心轮函数 `cmn()` 中，`undefined + 轮常量t` 会产生 `NaN`，经 `&0xFFFFFFFF` 处理后被强制归零，导致**轮常量本身也被一并丢弃**（而不只是把消息字当作0处理）。该bug从最简单输入（如 `md5("a")`）开始就会产生与标准MD5完全不同的错误结果，是此前 `notify.js` 一直触发 `debug:bad_sign` 异常日志、被迫临时绕过签名校验的真正根因
- 修复方式：在构建消息字数组前，按消息总分块数预先用 `0` 填满整个数组（`new Array(totalWords).fill(0)`），避免数组空洞参与运算
- 已用 Node内置 `crypto` 与 `spark-md5` 双重交叉验证修复结果，覆盖空字符串、纯ASCII、中文字符串、完整签名拼接串等场景，结果均与标准MD5完全一致
- 确认 `notify.js` 中的ZPay签名验证逻辑本身从未被绕过、一直在正常拦截非法回调；此前因 `_md5.js` 计算错误，连正确的签名也会被误判为错误，现已随本次修复一并解决，无需改动 `notify.js`
