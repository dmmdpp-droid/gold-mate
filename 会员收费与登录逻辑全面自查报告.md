# 会员收费与登录逻辑全面自查报告

> 自查于 2026-07-01。逐条检查了 `index.html`、`pay.js`、`notify.js`、`sms-verify.js`、`sms-bind.js`、`member.js`、`_session.js`、`activate.js` 的完整链路，并对照订阅制产品行业最佳实践（Apple/Google官方开发文档、RevenueCat/Qonversion等专业信源）逐项核对。
> 发现按严重程度排序，🔴 为高优先级（涉及真实资金/客服风险），🟡 为中优先级（体验缺口），🟢 为低优先级（架构一致性建议）。

## 🔴 发现1（最高优先级）：支付成功但回调延迟超过2分钟时，用户可能彻底无法完成注册

### 问题描述

`checkPaymentResult()` 轮询支付结果，最多轮询2分钟（24次×5秒）。超时后的处理是：

```js
} else {
  _polling = false;
  localStorage.removeItem('pending_order');
  localStorage.removeItem('pending_plan');
  // 超时：关闭等待弹窗，恢复正常付费墙
  hidePaywall();
}
```

**问题**：ZPay官方文档（见仓库内《ZPay官方开发文档存档.md》）明确写明，如果第一次回调失败，ZPay会按 **0/15/15/30/180/1800/1800/1800/1800/3600秒** 的节奏重试，最长间隔可以到**半小时到一小时一次**。也就是说，"2分钟内没等到结果"是完全正常、真实会发生的场景，不是异常情况。

但当前代码在超时时**直接删除了`pending_order`**——这是前端唯一记录"我刚才付了哪笔订单"的地方。删除之后：

- 用户看到等待弹窗突然消失，**没有任何提示**，很可能以为支付失败了
- 即使用户后来想起来去点"已有会员？手机号登录"手动登录，`submitPhoneLogin()`里`orderNo`的取值逻辑是：

  ```js
  const orderNo = forceAfterPay ? (localStorage.getItem('_pending_order') || '') : '';
  ```

  只有`forceAfterPay`为true（即支付成功后的强制登录场景）才会带上订单号，普通手动点开的登录**永远传空字符串**。而`_pending_order`（注意没有下划线开头的`pending_order`已被删除）本身也只有在`handlePaymentSuccess()`里才会被设置——也就是说，只有轮询**真正成功**过一次，`_pending_order`才存在。超时场景下这个值根本没被设置过。

**后果**：如果ZPay回调延迟超过2分钟才到达，钱已经实际扣了、服务端的`order`/`result`记录最终也会正确生成，但**用户没有任何UI路径能把这笔已经成功的购买跟自己的手机号关联起来**——因为唯一的订单号线索被前端自己删除了，走"手动登录"也传的是空字符串。用户唯一的办法是联系开发者人工处理。

### 已有的恢复机制为什么没生效

代码里其实已经设计了两处"自动恢复"逻辑：

```js
// 页面加载时
if (localStorage.getItem('pending_order')) { checkPaymentResult(); }

// 用户从微信/其他App切回来时
document.addEventListener('visibilitychange', function() {
  if (document.visibilityState !== 'visible') return;
  const pendingExpiry = localStorage.getItem('_pending_expiry');
  if (pendingExpiry && !isPaid()) { showPhoneLogin(true, label); return; }
  if (localStorage.getItem('pending_order')) { checkPaymentResult(); }
});
```

这两处逻辑本身设计是对的——只要`pending_order`还在，用户重新打开/切回页面就会自动继续轮询。**问题就出在超时分支把`pending_order`删掉了，相当于亲手把这套恢复机制的地基拆掉了。**

### 建议修复方向（供讨论，暂未实施）

超时时**不要删除`pending_order`**，只停止当前这一轮高频轮询（避免无意义地每5秒打一次接口），改为显示一条明确提示，例如："支付确认可能需要更长时间，请稍后重新打开本页面查看，或用刚才的手机号走"已有会员登录"查询结果"。这样上面提到的两处已有恢复机制（页面重开、切回页面）会自然接力，不需要额外开发新功能。

同时`submitPhoneLogin()`里`orderNo`的取值，建议不要严格限定在`forceAfterPay`场景，只要`localStorage`里`pending_order`（或`_pending_order`）还有值，普通手动登录也应该带上，这样即使用户是隔了很久才想起来手动去登录，也能把订单核验上。

## 🔴 发现2：`_pending_expiry` 与 `_pending_expiry_hint` 命名不一致，导致"强制重新登录"功能从未生效过

### 问题描述

```js
// 写入（第1894行）
localStorage.setItem('_pending_expiry_hint', expiry.toString());

// 读取（第2305、2321行，出现两处）
const pendingExpiry = localStorage.getItem('_pending_expiry');   // 少了 _hint
```

全文搜索确认：**`_pending_expiry`（不带`_hint`后缀）这个key，从未在任何地方被写入过**，只有读取。这意味着"用户支付成功但没走完手机号登录，下次打开页面/切回页面时自动强制弹出登录框"这个专门设计的恢复功能，从写下这行代码的第一天起就是**死代码，永远不会触发**。

这是一个纯粹的命名笔误（多了/少了`_hint`四个字符），不涉及逻辑设计问题，修复起来很简单——但因为它悄悄失效，一直没被发现。

### 建议修复方向

把读取的两处key名改成`_pending_expiry_hint`，跟写入保持一致。同时因为发现1和发现2实际上是同一条恢复链路上的两个坑，建议一起修。

## 🟡 发现3：会员到期后，跟"从未付费的新用户"收到完全相同的提示，没有专门的"续费"文案

### 问题描述——直接回应你的提问

`isTrialExpired()`分支里，无论用户是：
- (a) 真正第一次访问、7天免费试用用完的新用户，还是
- (b) **刚花钱买了会员、现在正常到期了的老用户**

看到的都是同一句：

```js
memberStatusEl.textContent = '试用已到期 · 点击升级';
```

**追根溯源**：`isPaid()`函数在判定到期时会立刻清空`gold_expiry`：

```js
if (now > parseInt(expiry)) {
  localStorage.removeItem('gold_unlocked');
  localStorage.removeItem('gold_expiry');
  return false;
}
```

清空之后，"这个人到底曾经是不是付费会员"这个信息在前端就**彻底丢失**了，走到`isTrialExpired()`判断时已经无从分辨，只能显示统一的"试用已到期"文案。

另外，`verifyMembership()`向服务端查询时，`member.js`其实**已经**在返回结果里区分了`reason:'expired'`（服务端确认这个人曾经是会员，只是过期了）和查无记录两种情况，但前端拿到这个`reason`字段后完全没有使用，两种情况都是同样的"清空本地状态"处理：

```js
} else {
  // 服务端明确返回"非会员"（无论是 expired 还是查无记录），
  // 都应清除本地状态...
  localStorage.removeItem('gold_unlocked');
  localStorage.removeItem('gold_expiry');
}
```

### 行业对照

这在订阅制行业里是个有正式名字的成熟概念，叫 **"Grace Period"（宽限期）** 和 **"Restore Purchase"（恢复购买）**。Apple和Google的官方开发文档都把这个当作**强制要求**——RevenueFlo的技术博客原话："Apple's checklist explicitly expects a restore option on any paywall. If a user reinstalls or switches devices, they need a way to restore an existing sub."（Apple的审核清单明确要求付费墙必须有恢复购买选项）。Google Play官方文档给的具体UI建议是："display the user's current expiration date and recurring price with a Reactivate button"（展示用户的实际到期时间，配一个"重新激活"按钮），而不是把老用户当陌生人对待。

### 建议修复方向（供讨论）

1. `verifyMembership()`拿到`data.reason === 'expired'`时，除了清空`gold_unlocked`/`gold_expiry`，额外记一个轻量标记（比如`gold_last_expiry`存最后一次到期时间），供后续文案使用
2. `isTrialExpired()`分支的文案，如果检测到这个标记存在，改成类似"会员已于X月X日到期，续费立即恢复"，而不是笼统的"试用已到期"，并且**跳过/不再重复给新的7天试用**（这个跟之前讨论过的"试用期误判"问题是同一个技术根源，建议合并解决）

## 🟢 发现4：激活码兑换路径完全绕开了Cookie/Session架构

### 问题描述

`redeemCode()` → 服务端 `/api/activate` 验证通过后，前端直接调用本地的`unlock(days)`：

```js
function unlock(days) {
  const expiry = Date.now() + days * 24 * 60 * 60 * 1000;
  localStorage.setItem('gold_unlocked', '1');
  localStorage.setItem('gold_expiry', expiry.toString());
  ...
}
```

这条路径**完全没有创建`phone:{phone}`或`session:{sessionId}`记录**，纯粹是本地localStorage状态。也就是说，通过激活码解锁的用户，完全没有享受到V1.8.8刚做的Cookie持久化架构的保护——一旦这台设备的localStorage因为Safari ITP等原因被清空，这类用户**没有任何找回路径**，连手机号登录都用不了（因为服务端压根没有这个人的手机号记录）。

### 建议（优先级低，因为该功能目前隐藏）

这个入口目前是`display:none`隐藏状态，不是当前主要转化路径，可以先不处理。但如果以后重新启用激活码功能，建议在`activate.js`验证通过后也顺带创建/关联`phone`记录（如果用户当时也提供了手机号的话），或者在UI上明确告知"激活码兑换的会员，建议同时绑定手机号，否则换设备后无法找回"。

## 小结：与本次提问的直接对应关系

你问的"会员到期之后，是不是应该弹出续费的界面"——**目前的答案是：不会弹出专门的续费界面，会跟全新用户一样看到笼统的"试用已到期"提示**，这正是发现3描述的问题，行业标准做法（Grace Period + Restore Purchase）也印证了这确实是个值得补的缺口。

而在自查这条链路的过程中，进一步发现了一个更紧急的问题（发现1+2）——**如果ZPay的支付确认延迟超过2分钟，用户可能真的没法完成注册**，这个建议优先处理，因为它涉及真实的资金和客服纠纷风险，比"文案不够贴心"更严重。
